# Feature Specification: ワークフローを登録し、ノード接続を検証できるようにする

**Feature Branch**: `002-workflow-builder`

**Created**: 2026-07-21

**Status**: Draft

**Input**: User description: "ワークフローを登録する画面を作成する。ワークフローは、定義済みのActionを組み合わせて作成する。ワークフローは、ノードをつなげて定義できるようにする。作成したワークフローの定義をバリデーションチェック処理を入れる。"

## Clarifications

### Session 2026-07-21

- Q: 保存後のワークフローはどう扱うか → A: 保存後は不変にし、新しい版として登録する
- Q: 分岐と合流をどう扱うか → A: 分岐と合流を許し、循環だけ禁止する
- Q: ノードは何を保持するか → A: ノードは Action 参照と表示用ラベルだけを持つ
- Q: 保存形式はどうするか → A: ノードと接続を別々に正規化して保存する
- Q: 保存時の検証結果はどう返すか → A: 保存時は共通の検証結果だけを返す

## User Scenarios & Testing *(mandatory)*

### User Story 1 - ワークフローを作成する (Priority: P1)

利用者がワークフロー名を付け、登録済みの Action を選んでノードを配置し、開始から終了までの流れを画面で組み立てられる。

**Why this priority**: ワークフロー画面の中心価値であり、Action を組み合わせて処理を表現する出発点になる。

**Independent Test**: ノードを追加し、Action を選び、接続して、編集内容が 1 つのワークフローとして保存対象になることを確認できる。

**Acceptance Scenarios**:

1. **Given** 登録済み Action が存在する, **When** 利用者がワークフロー名を入力し、ノードを追加して接続する, **Then** ワークフロー定義として画面上に組み立てられる
2. **Given** ワークフローの編集画面が開いている, **When** 利用者が開始ノードと終了ノードを含む経路を作成する, **Then** その経路が 1 つのワークフローとして表現される

---

### User Story 2 - ワークフロー定義を検証する (Priority: P1)

利用者が保存前にワークフロー定義を検証し、不正な接続や不足を検出できる。

**Why this priority**: 不正なワークフローを保存すると、実行時に失敗しやすくなるため、登録と同じくらい重要である。

**Independent Test**: 不正な接続や不足がある定義を入力したときに、保存されず、どこが不正か確認できる。

**Acceptance Scenarios**:

1. **Given** 開始ノードが存在しない, **When** 利用者が検証または保存を行う, **Then** ワークフローは不正として扱われる
2. **Given** 終了ノードが存在しない, **When** 利用者が検証または保存を行う, **Then** ワークフローは不正として扱われる
3. **Given** ノード同士の接続に循環がある, **When** 利用者が検証を行う, **Then** 循環があることが示される
4. **Given** どこにも接続されていないノードがある, **When** 利用者が検証を行う, **Then** 未接続ノードが示される
5. **Given** 許可されていない Action を選んでいる, **When** 利用者が検証を行う, **Then** 不正な Action として示される

---

### User Story 3 - 保存済みワークフローを見直す (Priority: P2)

利用者が保存済みワークフローを読み込み、後から内容を確認して再編集できる。

**Why this priority**: 実運用では一度作ったワークフローを見直す場面があるため、作成後の運用性を高める。

**Independent Test**: 保存済みのワークフローを再表示し、ノードと接続を確認できる。

**Acceptance Scenarios**:

1. **Given** 保存済みワークフローがある, **When** 利用者が一覧から選ぶ, **Then** その定義内容が画面に表示される
2. **Given** 保存済みワークフローが表示されている, **When** 利用者が一部を変更する, **Then** 変更後の内容で再検証できる
3. **Given** 保存済みワークフローに変更を加える, **When** 利用者が保存する, **Then** 既存版は保持され新しい版として登録される

---

### Edge Cases

- 開始ノードと終了ノードが複数ある場合は保存できない
- 分岐ノードや合流ノードは許可し、循環だけを禁止する
- ノード数が増えても、接続の整合性が崩れないように扱う
- 参照先の Action が削除または無効化されている場合は不正として扱う
- 同じ Action を複数ノードで使う場合は、ノードごとに役割が分かるようにする
- 接続を途中で外したときは、孤立した経路を検出できるようにする
- ワークフロー名が空、または重複している場合は保存を止める

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 利用者はワークフローを画面から新規作成できなければならない
- **FR-002**: 利用者はワークフロー名を入力できなければならない
- **FR-003**: 利用者は登録済み Action の中からノードに割り当てる対象を選べなければならない
- **FR-004**: 利用者はノード同士を接続してワークフローの流れを定義できなければならない
- **FR-005**: システムは開始ノードと終了ノードを含むワークフローのみ保存対象にしなければならない
- **FR-006**: システムは接続されていないノードを検出しなければならない
- **FR-007**: システムは循環する接続を検出しなければならない
- **FR-008**: システムは未登録または利用不可の Action を参照するノードを検出しなければならない
- **FR-009**: システムは保存前にワークフロー定義を検証し、問題がある場合は保存を拒否しなければならない
- **FR-010**: システムは検証結果を利用者が理解できる形で表示しなければならない
- **FR-011**: システムは保存済みワークフローを再表示できなければならない
- **FR-012**: システムは保存済みワークフローを再編集して再検証できなければならない
- **FR-013**: システムはワークフロー定義を Action 定義の組み合わせとして扱わなければならない
- **FR-014**: システムはワークフロー定義の保存時に、ノード構造と接続構造の両方を確認しなければならない
- **FR-015**: システムは保存済みワークフローを上書きせず、新しい版として登録しなければならない
- **FR-016**: システムはワークフローの版ごとの差異を追跡できなければならない
- **FR-017**: システムは分岐および合流を含む接続構造を保存対象として扱わなければならない

### Key Entities *(include if feature involves data)*

- **WorkflowDefinition**: ワークフロー名、説明、保存状態、ノード群と接続群を持つ定義
- **WorkflowVersion**: 保存されたワークフローの 1 版。元の定義を保持し、変更時は新しい版として追加される
- **WorkflowNode**: ワークフロー内の 1 つの処理要素。参照する Action と表示用ラベルを持つ
- **WorkflowEdge**: ノード間の接続を表す要素。処理順序を示す
- **WorkflowValidationResult**: 検証の成否、問題箇所、利用者向けメッセージを持つ結果

## Assumptions

- ワークフローはノードと接続を別々に正規化して保存する
- 保存時の検証結果はワークフロー全体の共通結果として返す

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 利用者は 10 分以内に 1 つのワークフローを作成して保存対象にできる
- **SC-002**: 不正な接続を含むワークフローは 100% 保存前に検出される
- **SC-003**: 検証結果を見た利用者の 90% 以上が、どのノードまたは接続に問題があるか 1 回で特定できる
- **SC-004**: 保存済みワークフローの再表示は、利用者が 5 秒以内に内容確認を開始できる
- **SC-005**: ノード数が増えても、利用者は保存前に整合性の有無を確認できる
- **SC-006**: 保存後の変更は既存版を壊さず、新しい版として 100% 記録される

## Assumptions

- ワークフローの作成画面は既存の workflow 管理領域に追加する
- 利用者は事前に Action 定義を登録済みである
- 初期スコープでは、実行そのものではなく定義と検証に集中する
- モバイル特化の編集体験は初期スコープ外である
- 保存済みワークフローの詳細なバージョン比較は後続機能として扱う
- 保存済みワークフローの編集は新しい版を作る操作として扱う
